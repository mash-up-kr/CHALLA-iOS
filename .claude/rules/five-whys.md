# Five Whys — 버그 원인 분석 기법

버그·크래시·이상 동작의 원인을 분석할 때는 증상만 고치지 말고 이 기법으로 근본 원인까지 파고든다.

## 절차

1. **문제 정의** — 문제나 증상을 명확하게 진술
2. **"왜?"를 다섯 번** —
   - 왜 이 문제가 발생했는가? → 답변 1
   - 왜 답변 1이 발생했는가? → 답변 2
   - 왜 답변 2가 발생했는가? → 답변 3
   - 왜 답변 3이 발생했는가? → 답변 4
   - 왜 답변 4가 발생했는가? → 답변 5 (근본 원인)
3. **근본 원인 검증** — 논리 사슬을 검증하고, 근본 원인을 해결하면 재발이 방지되는지 확인. 근본 원인이 여러 개일 수 있음을 고려
4. **해결책 수립** — 증상뿐 아니라 근본 원인을 해결하고, 예방 조치와 시스템적 개선을 함께 마련

## 예시

**문제**: 대용량 파일을 처리할 때 앱이 크래시함

1. **왜?** → 메모리가 부족해짐
2. **왜?** → 파일 전체를 한 번에 메모리에 로드함
3. **왜?** → 파서가 streaming을 고려하지 않고 설계됨
4. **왜?** → 초기 요구사항에 작은 파일만 명시됨
5. **왜?** → 요구사항 수집 때 향후 성장을 고려하지 않음

**근본 원인**: 불완전한 요구사항 수집 프로세스
**해결책**: streaming 파서 구현 + 요구사항 프로세스 개선

## 원칙

- 사람이 아닌 프로세스에 집중한다
- 시스템적인 문제를 찾는다
- 분석 내용을 문서화한다 (이슈/PR에 남기기)
- 해결책이 근본 원인을 실제로 해결하는지 테스트한다
