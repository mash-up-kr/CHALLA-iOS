1하네스 동작 원리 — 로드 계층

지시를 어디에 두느냐가 곧 언제 읽히느냐입니다. 아래로 갈수록 "필요할 때만" 로드됩니다.

항상 로드
CLAUDE.md — 프로젝트 첫 페이지. 빌드 명령, 모듈 의존 규칙 여섯 가지, 커밋 컨벤션, 에이전트 호출 안내 (루트, 200줄 미만 유지)
경로 매칭 시
.claude/rules/*.md — 해당 경로의 파일을 만질 때만 로드되는 세부 규칙 (예: Feature 코드 수정 시 TCA 규칙 발동)
관련 작업 시
.claude/skills/ — 작업 방법을 담은 참고 문서. 관련 작업이 시작되면 자동 로드 (평소엔 한 줄 설명만 올라와 있어 가벼움)
링크 참조 시
docs/*.md — 배경 설명 원본. CLAUDE.md·rules가 "상세는 여기"라고 가리킬 때만 읽음 (ARCHITECTURE·CONVENTIONS·DESIGN_SYSTEM·DEPLOYMENT·AI_WORKFLOW)
위임 시
.claude/agents/ — 역할 전용 일꾼. 별도 기억 공간에서 일하고 결과만 반환 (자동 위임 또는 이름 지명)
도구 사용 시
.claude/settings.json — 팀이 합의한 권한 목록. 빌드·조회 명령은 확인 없이 실행되고, 커밋·푸시·PR 생성은 항상 사용자 확인을 거칩니다 (허용 목록 전체는 8번 참고)
향후 도입
Hook — 특정 시점에 무조건 실행되는 검사 스크립트 (후보 검토 중 — 커밋 전 MODULE.md 동기화 검사 등)
핵심 원리: 항상 필요한 건 위에(얇게), 가끔 필요한 건 아래에(조건부로). 이렇게 나눠야 Claude의 기억 공간이 낭비되지 않고, 긴 세션에서도 규칙이 안 잊힙니다.

2개발 파이프라인 흐름

이슈 하나를 처리하는 표준 순서. 원칙: 설계 없이 구현으로 직행하지 않는다.

1
탐색 — swift-search
"관련 코드가 어디 있지?" — 직접 grep 하지 말고 위임. 검색 노이즈로부터 메인 대화를 보호
2
설계 — tca-architect / swift-architect
구현 전 State·Action·의존성·파일 구조 설계. 시안 분석이 필요하면 swift-ui-design을 먼저
3
구현 — tca-engineer / swift-engineer
TCA 리듀서는 tca-engineer, Domain·Data·Core 등 순수 Swift는 swift-engineer
4
뷰 — swiftui-specialist
코어 로직이 끝난 뒤 SwiftUI 뷰 구현. 색·폰트는 반드시 디자인 토큰 사용
5
리뷰 — swift-code-reviewer
커밋 전 셀프 리뷰. CHALLA 레이어 위반(Feature→Data import 등) 필수 검사 포함
6
테스트 — swift-test-creator
Swift Testing(@Test·#expect)·TestStore 기반. 모듈 내부 Tests/에 배치
7
문서 — swift-documenter
MODULE.md(책임·공개 API·의존성) 갱신, 주석 정리
✓
커밋 → PR
커밋·푸시는 항상 사람이 확인 버튼. PR에 Resolved: #이슈번호 → 리뷰 봇 + 팀원 리뷰 → 머지
문서 수정처럼 가벼운 작업은 중간 단계를 생략해도 됩니다. 단, 새 Feature는 2~7 전부 거치는 것이 기본입니다.

표에서 눈여겨볼 패턴 세 가지:

  • modern-swift는 거의 모든 에이전트가 참고합니다 — Swift 6.2 strict concurrency를 쓰는 프로젝트라, 코드를 직접 작성하는 아홉 에이전트가 전부 이 스킬을 봅니다. 코드를 쓰지 않고 찾기만 하는 swift-search만 예외입니다.
  • composable-architecture는 TCA 작업을 맡는 에이전트들만 참고합니다 — 설계하는 tca-architect, 구현하는 tca-engineer, 리뷰어, 테스트 작성자가 모두 같은 TCA 패턴 문서를 보고 일합니다. 덕분에 설계 따로 구현 따로 어긋나는 일이 생기지 않습니다.
  • 다섯 스킬을 전부 참고하는 에이전트는 swift-code-reviewer 하나뿐입니다 — 동시성부터 TCA, UI, 진단, 테스트까지 모든 영역을 검사하는 마지막 관문이기 때문입니다.

표에 없는 스킬 2종은 에이전트 없이 단독 동작: swiftui-specialist(스킬)는 메인 대화에서 SwiftUI 리뷰·deprecation 점검 시 발동, device-interaction은 시뮬레이터 조작 도구 설치 후 사용 가능(현재 대기).

4실전 사용 예시 — "방 생성 화면" 이슈를 받았다면

에이전트 이름을 몰라도 알아서 위임되지만, 지명하면 100% 확실합니다.

1. "RoomCreateFeature 관련해서 기존에 비슷한 화면 구조 있는지 찾아줘"
     → swift-search가 탐색

2. "이슈 #23 방 생성 화면 만들 건데, tca-architect로 설계부터 해줘"
     → State/Action/의존성/네비게이션 설계안이 나옴

3. "설계대로 구현해줘"
     → tca-engineer가 리듀서, swiftui-specialist가 뷰 구현

4. "커밋 전에 리뷰하고 테스트 만들어줘"
     → swift-code-reviewer → swift-test-creator 순서로 실행

5. "MODULE.md 갱신하고 커밋 준비해줘"
     → swift-documenter 정리 후, 커밋은 확인 프롬프트에서 승인

CLAUDE.md(항상 로드)는 이 모든 것의 첫 페이지입니다: 프로젝트 개요 → 빌드 명령 → 아키텍처 요약 → 컨벤션 → 에이전트 호출 가이드 → 테스트·MODULE.md 정책 → 개인 설정 안내. 규칙의 배경 설명은 docs/ARCHITECTURE.md·docs/CONVENTIONS.md·docs/DESIGN_SYSTEM.md가 원본이고, 배포 전략은 docs/DEPLOYMENT.md에 있습니다.

8권한 · 개인화 · 금지사항

팀 공유 권한 파일(.claude/settings.json)에 등록된 명령은 매번 묻지 않고 바로 실행됩니다. 목록에 없는 명령은 실행 전에 사용자에게 확인을 받습니다.

묻지 않고 실행되는 것구체적인 명령
빌드 · 시뮬레이터mise install/exec, tuist 전체, xcodebuild 전체, xcrun simctl 전체, swift build/test
git — 상태 조회git status, log, diff, show, branch, fetch
git — 커밋 전 준비git add, checkout, switch (여기까지만 — 커밋은 아래 참고)
GitHub 조회gh issue/pr의 list·view·diff·checks, gh label list, gh auth status
항상 확인을 받는 것: git commit, git push, gh pr create를 포함해 위 목록에 없는 모든 명령. 특히 커밋·푸시·PR 생성은 CLAUDE.md 컨벤션에서도 사용자 확인을 필수로 정해두었습니다. 이 파일에는 팀 공유 플러그인 4종(session-report · claude-md-management · serena · hookify)도 함께 등록되어 있어, 팀원 모두 같은 플러그인 구성으로 시작합니다.

개인 설정은 아래 파일로 분리합니다 — 전부 gitignore 대상이라 팀에 공유되지 않습니다.

하고 싶은 것방법
나만의 지시 (말투, 개인 습관)루트에 CLAUDE.local.md 생성 — gitignore되어 팀에 안 보임
나만의 권한 허용.claude/settings.local.json — 자동 생성·gitignore
내 모든 프로젝트 공통 설정~/.claude/CLAUDE.md, ~/.claude/skills/ (저장소 밖)
금지: 팀 공유 .claude/settings.json에 개인 토큰·API 키·개인 경로를 넣지 않습니다. 시크릿이 커밋에 섞이면 즉시 팀에 공유하고 키를 재발급합니다. 에이전트·스킬·rules 수정은 PR로 — 하네스도 코드처럼 리뷰받습니다.

9플러그인 — 하네스 확장 패키지

플러그인은 스킬·에이전트·훅·연동 서버를 한 묶음으로 설치하는 패키지입니다. 마켓플레이스에서 /plugin으로 설치하고, 팀 공유 목록은 .claude/settings.json에 기록되어 팀원 모두 같은 구성으로 시작합니다.

플러그인을 설치하면 하네스 방식이 바뀌나요? 아니요. 플러그인이 가져오는 것들은 1번에서 본 로드 계층의 같은 층에 그대로 편입됩니다 — 플러그인의 스킬은 "관련 작업 시" 층으로, 에이전트는 "위임 시" 층으로, 훅은 훅 층으로, 연동 서버(MCP·LSP)는 도구 목록으로 들어갑니다. 이름 앞에 플러그인이름:이 붙어서(예: session-report:session-report) 우리가 직접 만든 스킬과 충돌하지 않습니다.

팀 공유 플러그인은 네 개입니다. 순서에 의미가 있어요 — 먼저 재고(1), 고정비를 줄이고(2), 검색을 바꾸고(3), 실수를 차단(4)하는 흐름입니다.

1. session-report — 토큰 사용 명세서직접 호출
무엇을 주나
모든 세션 기록을 스캔해서 프로젝트별·스킬별·프롬프트별 토큰 사용량을 HTML 대시보드로 만들어 줍니다. "어디가 많이 먹었는지" 경고 카드까지 자동으로 채워줍니다.
언제 쓰나
비용이 어디서 새는지 모를 때 · 캐시가 잘 작동하는지 확인할 때 · 절약 작업의 전후를 비교할 때
사용법
/session-report 또는 자연어로 "지난 7일 세션 리포트 만들어줘"
테스트 보고 (정욱)
7일치 리포트를 직접 돌려봤다. 세션 하나가 주간 토큰의 66%(95.9M)를 차지하고 있었고, 그 안에서 짧은 질문 하나가 10.9M을 쓴 것까지 짚어줬다.
2. claude-md-management — CLAUDE.md 군살 관리직접 호출
먼저 알 것
CLAUDE.md는 모든 요청에 자동으로 따라붙는 파일입니다. 비대해질수록 매 요청 토큰이 늘어나는 고정비가 됩니다.
무엇을 주나
도구 두 개 — /revise-claude-md는 회의록(방금 끝낸 세션에서 배운 것을 추출해 추가 후보 제안), claude-md-improver는 감사관(모든 CLAUDE.md를 6개 기준 100점으로 채점하고 diff를 승인받아 반영).
언제 쓰나
의미 있는 세션을 마쳤을 때 회의록을, 분기·마일스톤마다 감사를. 평소에는 세션 중 # 키로 배운 것을 즉시 캡처하는 게 가장 쌉니다.
사용법
/revise-claude-md 또는 "CLAUDE.md 감사해줘"
테스트 보고 (정욱)
우리 프로젝트의 CLAUDE.md 전체를 검사시켜 봤다. 점수를 매겨주고(B등급) 개선할 부분을 짚어줘서, 그대로 반영했더니 A등급으로 올릴 수 있었다.
3. serena — 글자가 아니라 의미로 찾는 코드 검색자연어 요청
무엇을 주나
Claude Code에 코드 탐색기를 추가하는 플러그인입니다. Xcode로 치면 ⌘클릭 "정의로 이동"과 "참조 찾기"를 Claude에게 붙여주는 것. grep으로 login을 검색하면 loginMessage 변수와 주석 속 "로그인"까지 전부 잡히지만, serena는 loginTapped 액션이 실제로 정의되고 호출되는 곳만 짚습니다.
언제 쓰나
버그 추적으로 호출 지점을 하나도 빠짐없이 찾아야 할 때 · 이름 변경·리팩터링 · 1,000줄 파일에서 함수 하나만 필요할 때 (파일 통째 읽기 대비 토큰 수십분의 일)
사용법
자연어로 "LSP로 찾아줘" — 별도 명령은 없습니다
테스트 보고 (정욱)
같은 질문("이 심볼 어디서 쓰여?")을 두 방식으로 시켜봤다. swift-search는 약 23,400토큰을 써서 맥락 요약을, serena는 수백 토큰으로 참조 43곳의 줄·칸 위치를 돌려줬다.
준비물
uv 설치(brew install uv) + 빌드로 인덱스 생성 + buildServer.json(xcode-build-server). 셋 다 갖춰져야 참조 검색이 동작합니다.
4. hookify — 사고 치기 전에 멈추는 규칙규칙 0개 — 만들면 동작
무엇을 주나
반복되는 실수를 도구 호출 직전에 자동 검사해서 차단하는 훅을 마크다운 규칙 한 장으로 만들어 줍니다.
언제 쓰나
같은 지적을 두 번 이상 하게 될 때. "잘못 씀 → 지적 → 사과 → 재시도" 사이클은 회당 수천 토큰인데, 이걸 시작 전에 끊습니다.
사용법
/hookify node_modules에 파일 쓰지 마 한 줄이면 규칙 파일이 생성되고 다음 턴부터 즉시 적용(재시작 불필요). /hookify:list로 목록, /hookify:configure로 켜고 끄기.
주의
너무 넓은 패턴(예: log)은 login·dialog까지 막아 정상 작업을 방해합니다 — 단순하게 시작하세요. 현재 우리 규칙은 0개라 감시 지점만 있고 아무것도 막지 않는 상태입니다.

개인적으로만 쓰는 플러그인(발표자료 생성 등)은 팀 목록에 넣지 말고 개인 설정(.claude/settings.local.json)에 두세요. 설치·제거 후에는 /reload-plugins로 반영합니다.

serena와 swift-search는 뭐가 다른가

"loginTapped 어디서 쓰여?"라는 같은 질문을 둘 다 처리할 수 있어서 헷갈리기 쉽습니다. 정욱이 실제로 같은 질문을 둘 다에게 시켜서 비교한 결과입니다.

swift-search (서브에이전트)serena (LSP)
일하는 방식별도 대화 공간에서 grep으로 찾고 파일을 읽어서 요약을 만들어 돌려줌컴파일러가 만든 인덱스에서 심볼의 정의·참조 좌표를 바로 꺼내옴
정확도글자 일치 — login 검색이 loginMessage 변수, 주석 속 "로그인"까지 잡음loginTapped가 실제 정의·호출되는 곳만, 줄과 칸 위치까지
돌려주는 것맥락 해설이 담긴 요약 ("로그인 처리는 LoginFeature에 모여 있고 View는 액션만 보낸다")위치 목록만 — 해석 없음
비용 (정욱 실측)약 23,400 토큰 · 도구 호출 13회 · 약 69초수백 토큰 · 호출 1회 · 몇 초
준비물없음 — 즉시 동작빌드로 만든 인덱스 + buildServer.json(xcode-build-server) 필요. 최초 설치 시 uv도 필요(brew install uv)
결론 — 이럴 때 이걸 쓰세요.
· 로그인 버그를 추적하느라 loginTapped가 호출되는 곳을 하나도 빠짐없이 찾아야 할 때, LoginFeature의 이름을 바꾸는 리팩터링을 할 때 → serena
· "로그인 기능이 전체적으로 어떻게 구성돼 있어?"처럼 구조와 맥락을 이해해야 할 때 → swift-search
덧붙여 swift-search가 쓰는 토큰은 별도 공간에서 소비되어 메인 대화를 무겁게 하지 않고, serena의 결과는 메인 대화에 직접 쌓입니다 — 참조가 수백 개인 심볼을 조회할 땐 이 점도 고려하세요.

10원격 리뷰 봇 — 추후 업데이트 예정

PR을 올리면 GitHub Actions에서 Claude가 자동으로 코드를 리뷰합니다. 설정은 main 브랜치에 있습니다 — 실행 설정은 .github/workflows/claude-code-review.yml, 리뷰 기준은 Claude-Code-PR-Review/ 폴더의 문서들입니다.

상세 가이드는 추후 업데이트 예정입니다. 방향은 정해져 있어요: 로컬 점검(swift-code-reviewer)과 원격 점검(리뷰 봇)을 서로 독립된 관문으로 이중화해서, 한쪽 리뷰가 놓치거나 스스로 옳다고 믿어버리는 것(확증 편향)을 다른 쪽이 걸러내는 구조로 갑니다.

11한눈에 보기 — 절차와 하네스의 연결

개발 절차의 각 단계가 하네스의 어떤 부품을 쓰는지 한 장으로 정리했습니다. 단계나 부품에 마우스를 올리면 연결된 것만 밝게 남습니다.

읽는 순서는 위에서 아래입니다: 세션이 열리면 CLAUDE.md가 깔리고, 탐색·설계·구현·뷰·리뷰·테스트가 각자 필요한 부품을 쓰다가, 마지막 커밋·PR에서 권한 확인과 원격 리뷰 봇을 거쳐 끝납니다.

누가 무슨 말에 움직이나 — 질문 → 에이전트 → 스킬

왼쪽처럼 말하면 가운데 에이전트가 움직이고, 그 에이전트는 오른쪽 스킬을 참고해서 일합니다. 마우스를 올려 확인해보세요.
