의존성 규칙

단방향Feature → Domain ← Data. 상위만 하위를 참조
주입Feature는 Data를 import ✕ → DIContainer가 구현체 주입
화면 전환Feature끼리 직접 참조 ✕ → App이 TCA로 조립
NetworkData만 import. Feature·Domain은 서버의 존재를 모름
엔티티 단일화Room은 RoomDomain 1곳에만 정의 → 변환지옥 차단

레이어 판별 기준 (한 줄 규칙)

사이드 이펙트함수가 결과값 계산 이외에 바깥세상(OS·기기·서버)을 읽거나 바꾸는 것 — keychain.save() · logger.info() · camera.capture(). 순수 코드는 같은 입력이면 항상 같은 출력 · 바깥세상 불변
Shared시뮬레이터 없이 유닛테스트가 돌아가는 순수 코드. import Foundation만 허용 · 아무것도 import ✕
CoreOS를 만지는 코드 (Keychain·AVFoundation·os.log…). Shared import 가능
Network서버를 만지는 코드. Core(Logger 등)·Shared import 가능
토큰 흐름Network는 TokenProvider(protocol)만 정의 → AuthData가 Keychain으로 구현 → DIContainer가 연결. Network는 키체인의 존재를 모름
Core 명칭CoreData·CoreLocation처럼 Core = 저수준 기술 래핑이라는 Apple 관례. 순수 규칙은 Domain, 사이드 이펙트는 가장자리(Core·Network) 격리 — 배치 원칙은 Clean Architecture와 동일

모듈 세트 규칙 (테스트 · 문서)

한 세트모듈 하나 = Sources/ + Tests/ + MODULE.md. 구현만 있는 모듈은 미완성
테스트Swift Testing으로 작성. Domain은 순수 유닛테스트(가장 두텁게) · Feature는 TestStore · Data는 Mock 검증
MODULE.md책임 · 공개 API · 의존 관계 기록. 공개 API가 바뀌면 같은 PR에서 갱신
피처 데모앱새 Feature를 만들면 Demo/도 함께 — Mock 데이터로 그 화면만 단독 실행·검증
